PART 1: ANALYSIS OF DISCREPANCIES & MISTAKES
--------------------------------------------------------------------------------

1. CRITICAL FORMATTING ERROR: SPEAKER NOTES SHIFT (OFFSET BY 4 SLIDES)
   - Discrepancy: The speaker notes in the presentation file (.pptx) are shifted 
     forward by exactly 4 slides. The notes found on Slide 1 belong to Slide 5, 
     the notes on Slide 2 belong to Slide 6, and so on. Consequently, Slides 13 
     to 16 have no notes in the PPTX.
   - Correction: In this transcript, all speaker notes have been re-aligned to 
     their correct slides. Professional, relevant speaker notes have been drafted 
     for Slides 1 to 4 to complete the deck.

2. SLIDE 6: OS/ROS COMPATIBILITY WARNING
   - Discrepancy: Slide 6 states: "SYSTEM: UBUNTU 24.04 (LAPTOP SIDE) + ROS 2 
     HUMBLE HAWKSBILL."
   - Analysis: ROS 2 Humble Hawksbill officially targets Ubuntu 22.04 LTS (Jammy 
     Jellyfish). Ubuntu 24.04 LTS (Noble Numbat) officially runs ROS 2 Jazzy 
     Jalisco. Running Humble on Ubuntu 24.04 requires compiling from source or 
     running inside a Docker container, which is a non-standard setup. The project 
     documentation also lists Python 3.10, which is standard on Ubuntu 22.04 
     (Ubuntu 24.04 uses Python 3.12).

3. SLIDE 13 (NOTES SHIFTED TO SLIDE 9): OBSTACLE BYPASS DISTANCE MISMATCH
   - Discrepancy: The speaker notes state: "the robot successfully detected the 
     obstacle at the 0.85-meter warning distance, automatically selected the 
     clearest path, bypassed it..."
   - Code Check: In the code (code/bypass_follower_node.py line 49), the obstacle 
     trigger distance is set to 0.80 m (OBSTACLE_TRIGGER_DIST = 0.80). The 0.85 m 
     value is actually the target following distance (SAFE_DIST) used in the 
     separate behavior_tree_node.py. The slide notes should state 0.80 m.

4. SLIDE 11 (NOTES SHIFTED TO SLIDE 7): SIMPLIFIED SEARCH TRANSITION
   - Discrepancy: The notes say: "When the target is lost from the camera view, 
     the robot switches to SEARCH and rotates to locate the person again."
   - Code Check: In the behavior tree state machine (code/behavior_tree_node.py), 
     when the target is lost, the robot first transitions to REROUTE (where it 
     navigates to the last known position using Nav2 or manual vector guidance) 
     and only switches to SEARCH if it arrives or the reroute times out. The 
     notes simplify this transition and omit REROUTE.

5. SLIDE 15 (NOTES SHIFTED TO SLIDE 11): OSNET FUTURE WORK MISMATCH
   - Discrepancy: The notes list: "We also plan to integrate a stronger OSNet 
     Re-ID model, ideally utilizing a GPU or highly optimized CPU backends."
   - Code Check: In code/perception_node.py (lines 106-127), the code already 
     implements OSNet (osnet_x0_5 via PyTorch CPU) as its primary deep Re-ID 
     backend, falling back to MobileNetV3 only if OSNet/torchreid is not installed. 
     Thus, OSNet is already integrated as the preferred model.

6. PLACEHOLDER FOOTER DATE
   - Discrepancy: All original speaker notes begin with "1.7.2013", which is an 
     outdated placeholder date from a PowerPoint template. The project 
     documentation lists the correct date as "July 2026".

PROJECT OVERVIEW
03
A ROS 2 PERSON-FOLLOWING SYSTEM THAT IDENTIFIES ONE PERSON, FOLLOWS SAFELY, 
AVOIDS OBSTACLES, AND RECOVERS THE TARGET AFTER OCCLUSION.
- TARGET PERCEPTION: YOLO + BYTETRACK + RE-ID
- SAFE FOLLOWING: CAMERA + 360° LIDAR
- RECOVERY BEHAVIOR: REROUTE, SEARCH & REACQUIRE
- VALIDATION: GAZEBO + REAL ROBOT

SPEAKER NOTES (Drafted):
"To solve these challenges, we built a comprehensive person-following system in 
ROS 2. The core system consists of three main blocks. For target perception, we 
use YOLOv8 for person detection, ByteTrack for temporal tracking, and a custom 
multi-level Re-ID pipeline to prevent ID-swaps. For safe following, we combine 
camera bearing data with a 360-degree LiDAR range finder to steer and regulate 
linear speed. For recovery, we implemented a state machine that handles rerouting 
to the last known position, performing a 360-degree search spin, and executing a 
wide sweep to reacquire the target if lost. Finally, we validated our algorithms 
extensively in Gazebo before deploying them on the physical TurtleBot3 hardware."

SLIDE 7: SYSTEM ARCHITECTURE
================================================================================
SLIDE TEXT:
SYSTEM ARCHITECTURE
06

SPEAKER NOTES (Re-aligned from Slide 3):
"This slide shows the overall ROS 2 system architecture. The RGB camera and 
360-degree LiDAR send visual and distance information to the perception node. 
The perception node detects and tracks the selected person, then publishes the 
target information through the /tracked_target topic. This includes the 
target’s distance, angle, and tracking status—visible, occluded, or lost. 
The behavior tree node receives this target data, together with the LiDAR scan 
and robot odometry, and sends movement commands through /cmd_vel to the 
TurtleBot3 Waffle. The manager node controls the startup sequence: it waits for 
the system to initialize, sends the TRAIN command, waits until the target is 
visible, and then sends START. Finally, the dashboard monitors the camera, 
LiDAR, target status, robot state, odometry and logs, and also provides an 
emergency stop. So, in simple terms, perception determines who and where the 
target is, behavior decides what the robot should do, and the manager controls 
when the system begins."

SLIDE 8: PERCEPTION MODULE
================================================================================
SLIDE TEXT:
PERCEPTION MODULE
07

SPEAKER NOTES (Re-aligned from Slide 4):
"This slide explains the internal pipeline of the perception node. First, the 
RGB camera continuously sends images to the system. YOLO processes every image 
and detects all people visible in the scene. However, YOLO only detects the 
class ‘person’; it does not remember which person is our selected target. 
Therefore, we use ByteTrack. ByteTrack associates detections between 
consecutive video frames and assigns tracking IDs. This allows the system to 
follow the movement of each detected person over time. After that, we perform a 
40-frame enrollment phase. During this phase, the selected person stands near 
the centre of the camera, and the system collects appearance information from 
multiple frames. Using 40 frames is more reliable than using only one image 
because the person may move, turn, or appear differently between frames. The 
stored appearance information is then used for re-identification, or Re-ID. 
Re-ID helps the system recognise the same selected person after an occlusion or 
after ByteTrack changes the person’s tracking ID. This prevents the robot from 
accidentally switching to another person. The target’s angle is estimated 
from their horizontal position in the camera image. If the person appears on the 
left, the robot needs to turn left; if they appear on the right, it turns right. 
The distance is mainly estimated using the LiDAR measurement in the direction 
of the detected person. Finally, the perception node publishes /tracked_target, 
containing target visibility, distance, angle and tracking information. It 
also publishes /yolo/debug_image, which is the camera image with bounding boxes 
and labels, allowing us to visually verify that detection and tracking are 
working correctly. The key difference is that YOLO detects people, ByteTrack 
follows detections between frames, and Re-ID confirms that the detected person 
is the same person we enrolled."

SLIDE 9: BEHAVIOR AND NAVIGATION
================================================================================
SLIDE TEXT:
BEHAVIOR AND NAVIGATION
08

SPEAKER NOTES (Re-aligned from Slide 5):
"The Behavior node translates perception into physical action. We designed a 
state machine consisting of WAIT, FOLLOW, AVOID, REROUTE, SEARCH, and ACQUIRE 
states. When following, the robot uses a Proportional controller to smoothly 
adjust its linear and angular velocity to maintain distance. When an obstacle 
is detected, the robot smoothly transitions to the AVOID state. Here, we utilize 
a potential-field method that blends 70 percent obstacle repulsion with 30 
percent target attraction, allowing the robot to steer around the blockage while 
maintaining its general heading toward the owner."

SLIDE 14: CHALLENGES
================================================================================
SLIDE TEXT:
13 CHALLENGES :
- DYNAMIC OBSTACLES AND OCCLUSIONS IN CLUTTERED ENVIRONMENTS.
- LIGHTING VARIATIONS AFFECTING CAMERA-BASED DETECTION AND HSV.
- LIMITED ONBOARD COMPUTING (NO GPU) REQUIRING EFFICIENT MODELS AND CAREFUL 
  CPU USAGE.
- HARDWARE ISSUES SUCH AS LIDAR QOS MISMATCHES AND MOTOR BROWNOUTS DURING 
  AGGRESSIVE MANEUVERS.

SPEAKER NOTES (Re-aligned from Slide 10):
"Deploying this system introduced several engineering challenges. Dynamic 
obstacles and occlusions in cluttered environments frequently tested our 
tracking logic. Furthermore, lighting variations heavily impacted the 
camera-based detection and our HSV color histograms. Hardware limitations were 
also a major factor; without a GPU, we had to heavily optimize our deep 
learning models for CPU inference. Finally, we had to overcome 
hardware-specific hurdles, such as QoS mismatches dropping our LiDAR data and 
the aforementioned motor brownouts during sharp maneuvers."
